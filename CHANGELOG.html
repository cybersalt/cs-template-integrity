🔒Security

Eight findings from the v2.0–v2.2 retroactive review closed in this release:

  • H-1 — Joomla API token plaintext exposure (privilege escalation). A user with cstemplateintegrity.view could read the higher-privileged admin's saved Joomla API token off the dashboard's prompt block, and then authenticate against the entire Joomla Web Services API as the issuer (full privilege escalation outside cstemplateintegrity's scope). The dashboard view now substitutes the token only when the current user holds the write tier — view-tier users see the <PASTE YOUR JOOMLA API TOKEN HERE> placeholder.
  • M-1 — Backup restore path-safety hardening. v0.9.0 closed the live RCE primitive but the restore-side guard still permitted writes anywhere under JPATH_ROOT for any non-PHP extension — .htaccess, .user.ini, or template .css/.js could be overwritten via a write-tier user with seeded archival backup rows. PathSafetyHelper::assertOverrideWriteAllowed() is now a positive allow-list: every write must terminate inside <root>/templates/<tpl>/html/ or <root>/administrator/templates/<tpl>/html/ regardless of file extension.
  • M-2 — Disk-fill primitive on apply_fix and restore. Neither write path capped the size of the new contents being written. New 4 MB upper cap enforced before file_put_contents() in both paths.
  • M-3 — apply_fix silently failed on >1 MB legitimate overrides. Backup row size cap raised from 1 MB to 4 MB so apply_fix can both snapshot and write on real-world large overrides.
  • L-1 — Anthropic API key fingerprint narrowed brute-force search. Diagnostics fingerprint previously rendered starts="<first 8>", ends="<last 4>" of the saved key. Replaced with a 12-char SHA-256 prefix — admins can still verify "this is the key I think it is" but no chars of the actual key leak.
  • L-2 — Web Services apply-fix accepted nonexistent session_id. Audit-trail integrity bug. The endpoint now validates session_id references an existing session row; non-existent ids return 400 INVALID_SESSION_ID.
  • L-3 — Defense-in-depth on simplexml_load_file(). Now passes LIBXML_NONET explicitly — belt-and-braces for older PHP and future libxml default changes.
  • L-4 — Per-user soft cap on automated scans. Run automated scan now refuses if the current user has logged 12+ scan attempts in the past hour. Defends against accidental click-spam and against a CSRF-coerced admin (or compromised account) burning the saved Anthropic key's spend quota.
  • L-5 — Session-download filename consistency. A session named .. or . previously produced cosmetically odd download filenames. Sanitization now strips leading/trailing dots and falls back to session-<id> when empty.

📝Docs

  • Joomla Brain UI patterns — new section documenting the Cybersalt action-button palette, role/class/color table, "avoid" list, and CSS recipes for the orange Method 1 button and the mode-aware Rescan button. Future Cybersalt extensions can copy-paste from there.
  • Joomla Brain company-info — new "cs prefix vs. 'Cybersalt' full name" section so the next extension doesn't drift between the technical short form (in code) and the brand name (in user-facing strings).
  • Joomla Brain version-bump checklist — security review is now a hard prerequisite before any version bump. Routine bumps use the 0.7 confidence threshold; major bumps drop to 0.5 and require PHPStan in addition.

Migration

In-place upgrade from 2.1.x. No schema changes; no settings changes. Backup-row size cap bump from 1 MB to 4 MB doesn't affect existing rows. Side menu reorder takes effect on next admin cache clear. View-tier users who previously saw the saved Joomla API token in the dashboard's prompt block will now see the placeholder instead — restore the convenience by ensuring those admins also hold cstemplateintegrity.write (or core.manage).

v2.1.0 2026-04-29

🚀New

  • Choose your Claude model for the automated workflow. Two new dropdowns under Components → CS Template Integrity → Options → AI provider:
    • Scan model — for the one-shot Run automated scan. Default Sonnet 4.6. Haiku cuts cost ~3× with no real quality loss on classification; Opus is overkill for routine scans but useful for security-sensitive sites.

  • Cost-warning note below the dropdowns with rough per-scan dollar estimates and links to Anthropic's pricing + usage console.

🔧Improvements

  • Diagnostics → Run test uses Haiku unconditionally now (cheapest model) regardless of scan/chat settings — no reason to burn Sonnet/Opus tokens on a 16-token verify call.

🔍Security

  • Model-id values from #__extensions.params are validated against the three-entry whitelist before being passed to the Anthropic API. If a future config-form bypass somehow stored an unexpected model id, calls fall through to Sonnet rather than echoing arbitrary user-controlled strings into the request payload.

v2.0.0 2026-04-29

The whole automated-review-with-Claude path landed. v1.x exposed override data via REST and let your Claude session call back to apply fixes. v2.0 closes that loop: save your Anthropic API key, click Run automated scan, and the extension drives the entire review server-side — fetches every override, calls Claude, gets back a markdown report, saves it as a session. From the session view you can chat with Claude inline and have it apply fixes, dismiss findings, or clear the tracker.

🚀New

  • Run automated scan — one click on the dashboard drives the entire override review server-side using your saved Anthropic API key. No copy-paste prompt required. Caps at 60 overrides per call to stay within Anthropic's per-minute input-token budget.
  • Chat with Claude on the session detail view. Submit a follow-up like “fix #1 and #3, dismiss the cosmetic ones” and the server runs an Anthropic tool-use loop with six tools wired to existing helpers (apply_fix, dismiss_override, dismiss_all, etc.). Conversation persists turn-by-turn in a new messages column. Auto-backups still apply; everything Claude does is reversible.
  • Component Options dialog on the dashboard toolbar — one field for now (Anthropic API key), with the standard Joomla Permissions tab.
  • Diagnostics modal with API-key fingerprint, system info, and a one-click Run test button that fires a tiny say PONG request to verify the saved key works.
  • Loading-overlay system with spinner and elapsed-seconds counter. Locks the page during the 30–90s synchronous Anthropic calls so users don't navigate away or double-submit. Tracks Atum dark mode automatically.
  • First-run "Before you start" disclaimer modal — per-admin acknowledgment, persists in the action log, surfaces backup advice and Tim's contact email.
  • Sessions bulk download — tick rows in the session list and grab a single zip with one Markdown file per selected session.
  • Plain-English file-purpose descriptions on every backup row.

🔧Improvements

  • Dashboard layout reworked — quick-nav button row at the top, Reset overrides moved up from the bottom, both prompt cards now lead with an audience-naming alert banner.
  • Prompt boxes restyled with explicit dark-mode rules (#0d1117 background, GitHub-dark style), no-wrap so long lines scroll horizontally instead of leaving ragged hanging breaks, and a thicker border + slight inset shadow.
  • Disclaimer modal CSS now uses the dual [data-bs-theme="dark"] / [data-color-scheme="dark"] selector pattern plus a prefers-color-scheme: dark fallback — reads correctly across every Joomla 5/6 admin template's flavor of dark mode.
  • Dashboard version display reads from the on-disk component manifest at render time. No more hardcoded version strings going stale.
  • Top-level README rewritten with a prominent /releases/latest download link at the top.

📦Build

  • UTF-8 BOM detection in build-package.ps1. Fails the build with a list of offending files if any .php/.xml/.ini under packages/ has a BOM. Cannot ship a v1.0.2-style BOM regression again without going out of our way.
  • Schema migration in postflight — idempotent ALTER TABLE ADD COLUMN messages LONGTEXT runs on every install/update, gated by a column-exists check.
  • Joomla 5 + Joomla 6 native declared explicitly via <targetplatform name="joomla" version="5\.[0-9]+|6\.[0-9]+" /> in every manifest.

🐛Fixes

  • Disclaimer "don't show again" was failing silently — inline JS posted the form token via URL query string, but $this->checkToken() defaults to checking $_POST. Switched to $this->checkToken('get').
  • Anthropic API key was being truncated to 99 chars at save time. Joomla's password field type was the culprit; switched to type="text" with explicit maxlength="500" and filter="raw".
  • HTTP 400 "Input should be a valid dictionary" when chatting with Claude. PHP's json_decode($body, true) converts JSON {} to empty PHP arrays; re-encoding produces [], but Anthropic requires tool_use.input to be an object. ConversationRunner::fixEmptyToolInputs() now walks every assistant turn and replaces empty arrays with stdClass.
  • Anthropic 10K-tokens-per-minute rate limit blew on chat continue because persisted scan messages carried the full ~60K-token override-file dump. Replaced with a short summary + tool-fetch-on-demand model.
  • HTML entities in language strings (&mdash;, &ldquo;, &hellip;) replaced with literal UTF-8 characters; htmlspecialchars()-escaped strings were displaying &mdash; literally.
  • Dashboard prompt now lists write endpoints in the upfront Endpoints block with "THIS IS HOW YOU APPLY A PATCH" annotations — external Claude sessions had been reading the GET-only list and concluding the API was read-only.
  • Postflight no longer renders the install-success card on uninstall.

🔍Security

  • Every new write endpoint (display.runScan, display.testApiConnection, session.continueChat) is CSRF + ACL gated.
  • Tool-use fixes from chat run through the same path-safety guards as direct apply-fix calls — separator-anchored containment, PHP-extension whitelist, opcache_invalidate() after every PHP write.
  • Anthropic API key never leaves the server. Read from #__extensions.params per call, sent as x-api-key header, discarded.

⬆️Migration from 1.x

  • 1.0.x → 2.0.0 is an in-place upgrade. Schema migration is idempotent.
  • Sessions saved by the v1.0.4 build of the auto-scan had an oversized first user message; the first chat against one of those auto-rewrites it to a short summary on the fly.

v1.0.3 2026-04-29

🐛Hotfix

  • 1.0.2 produced an immediate "Compile Error: strict_types declaration must be the very first statement" 500 on every component view. The PowerShell pass during the 1.0.2 build saved seven template files with a UTF-8 BOM, which gets emitted as bytes before <?php parses and breaks declare(strict_types=1)'s "first statement" rule. All seven templates rewritten without BOM. Strongly recommend upgrading from 1.0.2 → 1.0.3 immediately; 1.0.2 is unusable.

v1.0.0 2026-04-25

🚀First stable release

Core loop — site owner pastes one prompt → Claude lists every flagged override on the site, classifies each one, writes a plain-English report, asks which to fix, applies the fixes the owner confirms, marks the rest as checked — has been live-tested end-to-end on a real client site running Joomla 6.1 on a third-party template.

📦What's in 1.0

  • com_cstemplateintegrity (admin) — dashboard, sessions log, action log, file backups with restore, gated by ACL actions in admin/access.xml.
  • plg_webservices_cstemplateintegrity — registers the /api/index.php/v1/cstemplateintegrity/* routes; auto-enabled by the package installer.
  • API endpoints: GET /overrides, /overrides/{id}/override-file, /overrides/{id}/core-file, /sessions/{id}; POST /overrides/{id}/apply-fix, /dismiss, /overrides/dismiss-all, /sessions, /backups/{id}/restore.
  • Auto-backup before every file write — every patch reversible from the admin UI or via the API.
  • Same-chat conversational flow as the default; cross-session fix prompt as the rare-case helper.
  • Plain-English descriptions on backup rows so non-technical users can tell at a glance what each file is.
  • Unknown-finding catch-all — if a finding doesn't fit "code change" or "configuration question", Claude stops and explains in plain English instead of attempting a partial fix.
  • claude.ai sandbox guidance built into the dashboard prompt (Settings → Capabilities → Code execution → Network access; close current chat and start a new one to pick up the updated allowlist).

🔍Security baseline

  • admin/access.xml with cstemplateintegrity.view / .write actions — every endpoint, admin and API, gated.
  • Separator-anchored path containment (no strpos prefix-collision bypass).
  • PHP-extension write whitelist — .php writes only allowed under templates/<tpl>/html/.
  • opcache_invalidate() after every PHP write so patches take effect immediately.
  • GET-form CSRF tokens on download links; CRLF/header-injection sanitization on Content-Disposition.
  • htmlspecialchars() on every Text::_() rendered into the installer frame.
  • No free-form file_path accepted in any request body — all paths derived server-side from #__template_overrides rows.
